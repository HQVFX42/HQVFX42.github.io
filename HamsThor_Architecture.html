HamsThor Spin to Win...ch!

모바일 세로 화면 하이퍼캐주얼 물리 게임. 앵커에 매달려 로프를 감아 회전 속도를 높이고(윈치), 원운동의 접선 방향으로 몸을 던져(릴리즈) 다음 앵커에 정밀하게 락온하며 무한히 높은 고도로 올라가는 것이 핵심 루프다.

PLAY DEMO (Itch.io)

모듈 구조

index.html 이 순서대로 로드하는 8개 스크립트가 전부다. 각 파일은 HamsThor 전역 객체에 자기 모듈만 등록하고, 서로는 그 공개 프로퍼티를 통해서만 접근한다.

1. config.js

중력, 감쇠, 윈치 지수, 락온 반경, 로비/튜토리얼 튜너블, altitudeDisplayScale 등 모든 게임 상수를 한 곳에 모은 CONFIG 객체.

2. level.js

무한 절차적 앵커/장애물 생성기. 카메라가 다가온 만큼만 위로 이어 붙이고(ensureGeneratedAbove) 화면 아래로 지나간 것은 가지치기(pruneFarBelow)해서 메모리를 고정 크기로 유지.

3. assets.js / sound.js

스프라이트/오디오 파일명 컨벤션 기반 자동 탐색 로더. 파일이 없으면 조용히 무시하고 벡터 폴백/무음으로 대체 — 에셋 유무에 관계없이 항상 실행 가능.

4. physics.js

순수 상태머신 HamsterController. DOM을 전혀 모르며 ORBIT / FLIGHT / DEAD 세 상태만 관리. 아래 물리 공식 참고.

5. input.js

마우스 휠 / 터치 드래그를 반지름 변화량(deltaR)으로, 탭을 화면 좌표로 변환해 콜백으로 넘기는 순수 입력 어댑터. 게임 상태를 모르고 호출만 한다.

6. flow.js

LOBBY / TUTORIAL_1-3 / TRANSITION / GAME 상위 FSM. main.js가 매 프레임 만들어주는 컨텍스트 객체를 통해서만 hamster와 카메라를 건드린다.

7. main.js

고정 타임스텝 루프, 카메라, 렌더링, HUD, 사망/공유 화면, 입력 디스패처 오케스트레이션. GAME 상태가 아니면 즉시 Flow에 위임하고 자기 로직은 건드리지 않는다.

2단 상태머신

본 게임 루프(HamsThor.STATE)와 온보딩 흐름(HamsThor.AppState)을 완전히 분리된 두 계층으로 둔 것이 이 아키텍처의 핵심 결정이다. 위 계층은 아래 계층을 그대로 재사용할 뿐, 아래 계층은 위 계층의 존재를 전혀 모른다.

상위 · flow.js

AppState

LOBBY → TUTORIAL_1 → TUTORIAL_2 → TUTORIAL_3 → TRANSITION → GAME

로비의 인터랙티브 펜듈럼 토이, 3단계 실전 튜토리얼(윈치 가속 → 관성 사출 → 정밀 락온), 그리고 본게임 진입 전 연출을 담당. 튜토리얼은 진짜 HamsterController 인스턴스를 던져쓰기(throwaway) 앵커 위에서 그대로 재사용한다 — 물리 공식을 다시 구현하지 않는다.

하위 · physics.js

STATE

ORBIT ⇄ FLIGHT → DEAD

실제 게임의 물리 상태만 관리하는 순수 상태머신. DOM도, AppState도, 카메라도 모른다. startOrbit / adjustRadius / release / latch / die 다섯 개의 공개 메서드가 전부 — 위 계층이든 실제 게임이든 이 다섯 개만 호출한다.

코어 물리 공식

① 윈치 — 각운동량 보존형 반지름-속도 트레이드오프

ω_new = ω_old · (R_old / R_new) ^ winchExponent

로프를 짧게 감을수록(R↓) 회전 속도(ω↑)가 빨라진다. 지수 winchExponent로 가속 곡선의 체감을 튜닝.

② 릴리즈 — 원운동 위치식의 미분(접선 벡터) 상속

vx = −R·ω·sin(θ)    vy = R·ω·cos(θ)

놓는 순간의 반지름·각속도·각도로부터 접선 방향 속도 벡터를 그대로 비행 초기속도로 상속한다. 상승 방향으로 놓을수록 콤보 배수가 붙어 vy에만 곱해진다(회전 방향엔 영향 없음).

③ 정밀 락온 — 외적(cross product) 기반 방향 판정

direction = sign(r × v)    ω0 = direction · max(|baseOmega|, minLatchOmega)

앵커 35px 반경 안에 탭이 들어오면, 진입 속도 벡터의 회전 "방향"만 취하고 크기는 항상 기준 각속도로 고정한다 — 어떤 궤적으로 들어와도 다음 회전은 일관된 속도로 시작하며, 어떤 설정 실수로도 ω가 0이 되지 않도록 최소 안전값을 보장한다.

고정 타임스텝 + 불릿타임

requestAnimationFrame 루프는 실제 경과 시간을 그대로 누산기(accumulator)에 쌓고, FIXED_DT(1/60) 단위로 소비하며 매 스텝 update()를 호출한다. 슬로우모션(불릿타임)은 누산기가 차는 속도를 늦추는 대신, 각 스텝에 넘기는 dt 자체를 축소 (FIXED_DT × Flow.timeScale)한다.

→ 이렇게 해야 물리 스텝이 매 프레임 정상적으로 돌면서 시뮬레이션 자체만 느려져, 진짜 슬로우모션처럼 보인다. (반대로 하면 스텝이 뜨문뜨문 돌아 프레임 드랍처럼 보이는 문제가 있었다.)
